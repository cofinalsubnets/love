ai for JS programmers

javascript has first-class functions, closures, and array methods. ai keeps all of that and throws out nearly everything else: no objects, no classes, no statements -- just values, and every value is a function you can call.

the one idea: a call is curried

a call is written prefix with no commas: (f x). multiple arguments are just nested calls -- (f x y) means ((f x) y) -- so partial application is automatic. there is no .bind and no partial; you just leave arguments off.

((\ x y (x + y)) 3 4)   ; 7        an arrow fn, defined and called
(map (+ 1) '(1 2 3))    ; (2 3 4)  (+ 1) is "add one" -- already partial

the JS you'd write is [1,2,3].map(x => x + 1). In ai the lambda is (\ x (x + 1)), and (+ 1) is the same function with one argument pre-supplied.

truthiness, but with one rule

js asks you to memorize the falsy values (0, "", null, undefined, NaN, false). ai has a single law: a value is false when it nets to nothing -- empty, zero, or negative. everything positive or non-empty is true.

(? ""  'yes 'no)   ; no
(? "x" 'yes 'no)   ; yes
(? 0   'yes 'no)   ; no
(? '(0 0) 'yes 'no); no      a list summing to 0 is nothing, too

numbers are functions

the strangest corner: a number is the function that applies its argument that many times. so calling a number is iteration -- and calling it on another number is exponentiation.

(2 (+ 1) 10)   ; 12   do (+ 1) twice to 10
(3 2)          ; 8    2 to the 3rd

1 is the identity function and 0 is "always 1". you rarely lean on this directly, but it's why arithmetic extends to functions: * of two functions is composition.

((* (+ 1) (+ 2)) 5)   ; 8    the same as (+1)∘(+2) applied to 5

gotchas

  • No operator precedence. Infix is purely right-associative: 2 * 3 + 1 is 2 * (3 + 1) = 8, not 7. When in doubt, parenthesize.
  • No chained comparison. 1 < 2 < 3 parses as 1 < (2 < 3) = false. Use (&& (1 < 2) (2 < 3)).
  • Integers stay exact. They don't overflow into floats -- they grow to bignums. (62 2) is 2⁶² to the digit.
  • No objects, no this. Where you'd reach for an object, reach for a book (map); it doubles as its own accessor function.

ai for Python programmers

python and ai are both dynamic, have exact big integers, and agree that "empty is false." the difference is that ai is fully curried and lisp-surfaced -- code is data, in parentheses -- and that its arrays ship with NumPy-style broadcasting built into the core.

the falsy rule you already trust

in python 0, "", [], {}, and None are false. ai takes that instinct and makes it the whole law: false is nothing -- any value that nets to zero or less. None corresponds to ().

(? ""  'yes 'no)   ; no
(? "x" 'yes 'no)   ; yes
(? () 'yes 'no)    ; no       () is ai's None
(? '(0 0) 'yes 'no); no       a list summing to 0 is empty-of-content

NumPy is already in the box

What you'd import NumPy for is in the core. @(...) is an array, and + - * / broadcast over arrays the way you expect.

(@(1 2 3) + @(10 20 30))   ; @(11 22 33)
(iota 3)                   ; @(0 1 2)        like np.arange(3)
(inner @(1 2 3) @(4 5 6))  ; 32              the dot product

currying is free partial application

Every function is curried, so functools.partial is just leaving arguments off the end. (+ 1) is a one-argument "add one"; pass it straight to map.

(map (+ 1) '(1 2 3))   ; (2 3 4)

big ints -- and a generic +

Like Python's, ai integers are exact and unbounded. But + and * are generic: + also concatenates strings and appends lists, and * repeats.

("ab" + "cd")     ; "abcd"
('(1 2) + '(3 4)) ; (1 2 3 4)
("ab" * 3)        ; "ababab"

gotchas

  • Chained comparison does NOT carry over. Python's 1 < 2 < 3 is True; ai parses 1 < 2 < 3 right-associatively as 1 < (2 < 3) = false. Write (&& (1 < 2) (2 < 3)).
  • No operator precedence. Infix is right-associative: 2 * 3 + 1 = 8, not 7. Parenthesize.
  • Everything is an expression. There are no statements -- a program reads more like one big comprehension than imperative Python.
  • Parentheses, not indentation. Whitespace is insignificant; the s-expression structure is everything.

ai for Scheme programmers

If you read Lisp, you can read ai: s-expressions, lambda, lexical closures, tail calls, call/cc, and macros are all here. Three things are new -- application is curried, numerals are Church numerals you can actually call, and +/* are generic -- and one thing is genuinely different: truth.

what's already familiar

(\ x (x + 1))           ; lambda
(cap '(1 2 3))          ; 1      car
(cup '(1 2 3))          ; (2 3)  cdr
(link 1 '(2 3))         ; (1 2 3) cons
(call-cc (\ k (k 41)))  ; 41     a one-shot call/cc

'() is written (); car/cdr/cons are cap/cup/link (with caap...cuup compounds, read right-to-left like the old c[ad]+r).

currying changes the feel

(f x y) is ((f x) y) and (f) is just f. So partial application is automatic -- (+ 1) is already a function, and (map (+ 1) '(1 2 3)) reads as plainly as it runs (it yields (2 3 4)).

numerals really are Church numerals

The Church encoding isn't a teaching trick here -- the numerals are the numbers, and applying one iterates its argument. Numeral-on-numeral is exponentiation.

(3 (+ 1) 0)   ; 3   apply (+ 1) three times to 0
(3 2)         ; 8   2 to the 3rd

And because +/* are generic over the whole value lattice, on functions * is composition:

((* (+ 1) (+ 2)) 5)   ; 8

truth is the one real difference

In Scheme only #f is false. In ai, false is nothing -- any value that nets to zero or less: (), 0, "", the empty list, even a list of negatives. if/cond test that net, not a boolean.

(? '() 'yes 'no)   ; no    -- '() is false here, unlike Scheme
(? 0  'yes 'no)    ; no

This is the place your Scheme reflexes will bite: (if '() 'yes 'no) is yes in Scheme, no in ai.

define / letrec / sequence, one form

: is letrec* + sequencing + define-sugar, all at once. Recursion just works; the last form is the value.

(: (fac n) (? (n < 2) 1 (n * (fac (n - 1)))) (fac 5))   ; 120

gotchas

  • Right-associative infix, no precedence. 2 * 3 + 1 is 2 * (3 + 1) = 8. The infix is a thin reader pass over the prefix forms, folding right.
  • An unbound name isn't an error. A name not in the global "book" reads () (nothing) rather than raising.
  • Quasiquote is inverted. The ` reader builds a list whose elements are all evaluated, so you quote the literal parts instead of unquoting the computed ones: `(1 (+ 1 1) 3) → (1 2 3).
